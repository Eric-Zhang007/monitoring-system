执行层总目标（一次性写对的验收口径）

最终你要得到：

确定性订单生命周期（OMS）：每个 decision / order / child_order / fill 都有可追溯的 DB 记录；状态机严格、可回放。

执行算法落地：至少支持 marketable_limit 与 passive_twap 两种执行风格（你 policy 已经在吐这两个 style）。

Live 适配器可用：Coinbase / Bitget 至少一个能实盘闭环（下单→部分成交→撤单→补单→最终状态），并且强 idempotency。

对账守护进程：自动修正“本地状态 vs 交易所状态”，异常触发 kill switch（你已有 risk_control_state）。

硬门禁测试：paper 模式下有集成测试验证：下单→执行→落库→PnL/仓位更新→trace 完整。

COMMIT 4 — ExecutionEngine：改成“OMS 驱动”的统一执行管线（paper/live 共用）

你现在 engine 有 adapter.execute(order, context) 返回一个 dict + lifecycle，但没有“分拆/子订单/落库/对账 hook”的统一编排。

4.1 新增 ExecutionEngine.run_decision（强制入口）

修改

backend/execution_engine.py

新增/重构主入口

run_decision(repo, decision_id, *, adapter, venue, market_type, max_orders, base_context) -> ExecuteOrdersResponse

run_decision 内部固定步骤（必须照抄成代码结构）

decision = repo.get_decision(decision_id)（若不存在创建）

拉取 orders：repo.fetch_orders_for_decision(decision_id)

对每个 parent order：

context = resolve_order_execution_context(order, base_context)（你已有）

生成执行计划 plan：plan = build_execution_plan(order, context, market_state)

plan 会产出 child_orders[]（分拆后的 IOC/GTC 限价单序列）

逐个 child order 执行（同 symbol 串行，跨 symbol 可并行但限并发）

每一步落库：

create_child_order row

submit -> update_child_status(submitted, venue_order_id)

poll -> fills -> insert_fills

cancel/replace -> lifecycle append

parent 聚合：update_parent_from_fills

decision 聚合：filled/rejected breakdown、latency、reject categories

4.2 新增执行计划器（Smart Order Router 的最小可用版本）

新增文件

backend/execution_planner.py

必须支持两种 style（对应你的 policy 输出）

marketable_limit：

使用 est_price + max_slippage_bps 计算可成交限价

先下 limit IOC/FOK（按 policy.tif），失败/部分成交 -> 允许补一个 market IOC（可选）

passive_twap：

将 qty 按 n_slices 分拆（比如 6~24 片，可配置）

每片用更保守的 limit（maker 偏移），间隔 sleep（在 live 模式）

注意：paper/backtest 不 sleep，但要模拟时间推进（记录 slice_index）

输入 market_state（最小可用）

从 DB 拉：最新 orderbook_l2（你 schema 有）

若拿不到 orderbook：退化为仅用 est_price（但必须在 trace 里标记 market_state_missing=1）

Done

planner 有单测：同样 qty+policy 必然产出稳定 child_orders（可复现）
